# 第11章：整洁架构（The Clean Architecture）

在上一章中，我们探讨了软件系统如何被组织成层，以将高层策略与低层细节分离。本章将更深入地了解整洁架构如何定义这些层、它们如何交互，以及如何构建一个遵循这些边界的系统——即我们的"CA 引擎"。

## 11.1. 依赖规则

整洁架构有一条规则，称为**"依赖规则（Dependency Rule）"**。该规则规定：系统中的依赖必须全部指向内部——从低层策略指向高层策略。

> 回顾问题：为什么高层策略不应该依赖低层策略？

这通常通过下面的同心环形图来可视化。最高层策略——实体（Entities）——位于架构的核心，而最低层策略——框架和驱动（Frameworks and Drivers）——位于架构的外部。

![](https://blog.cleancoder.com/uncle-bob/images/2012-08-13-the-clean-architecture/CleanArchitecture.jpg)

此时你可能会想：信息如何从内层流向最外层？这正是依赖反转原则（DIP）大显身手的地方！图中右下角的小图很好地总结了如何应用 DIP，在允许依赖指向内部的同时，允许信息从内部流向外部。我们在接下来更具体的 UML 类图中也会看到这一点。

## 11.3. 按层了解整洁架构

在完整讲解如何构建 CA 引擎并追踪控制流之前，让我们先快速了解构成 CA 引擎的所有组件。阅读每个组件时，请对照上面的图表。

### 11.3.1 框架和驱动（Frameworks and Drivers）

- **视图（View）**：显示信息并响应用户交互。View 会请求 Controller 执行用户想要的操作。View 显示的信息存储在 ViewModel 中。当 ViewModel 更新时，View 会收到通知并向用户显示新信息。

- **数据访问（Data Access）**：该类负责向程序外部的文件或数据库读写持久化数据。它是一个细节，实现用例层指定的数据访问接口（Data Access Interface）。它会创建用例（Use Case）用于完成工作的临时实体。

### 11.3.2. 接口适配器（Interface Adapters）

- **控制器（Controller）**：将原始用户数据转换为有用的形式（例如，将字符串转换为 Date 对象，或将浮点数转换为货币值），创建包含这些信息的输入数据（Input Data）对象，并调用方法启动用例，同时传入输入数据。

- **呈现器（Presenter）**：从用例交互器（Use Case Interactor）接收信息（输出数据对象），并将其转换为原始字符串和数字以供显示。Presenter 会用这些信息更新 ViewModel。

- **视图模型（View Model）**：这是一个存储 View 需要显示的信息的存储类。View 是否需要显示当前登录用户的名字？那么 ViewModel 就存储该名字。View 通过 ViewModel 访问它需要显示的任何信息。

### 11.3.3. 用例（Use Cases）

用例层包含用例交互器（Use Case Interactors），负责程序的主要操作。如果你的程序是日历应用，这一层负责创建和搜索事件；如果是消息应用，这一层负责创建新消息、存储消息、检索对话以及任何其他功能。这一层包含了应用程序用户能够做的一切事情的逻辑。

- **用例交互器（Use Case Interactor）**：接收输入数据，执行用例，在必要时通过数据访问对象查询信息并操纵实体。它可能会创建需要通过数据访问对象保存的新数据。完成后，创建一个输出数据（Output Data）对象——用例结果——并将其传递给 Presenter。

- **输入数据（Input Data）**：Controller 创建此对象，并将用户输入存储在其中，然后将其传递给用例交互器。它被设计为用例交互器完成工作时最便于使用的格式。

- **输出数据（Output Data）**：此类代表执行用例交互器的结果。用例交互器创建此对象，并通过输出边界（Output Boundary）将其发送给 Presenter。

该层内置的接口有：

- **输入边界（Input Boundary）**：由用例交互器实现。Controller 类调用此接口中的方法。

- **输出边界（Output Boundary）**：由 Presenter 类实现。用例交互器调用此接口中的方法。

- **数据访问接口（Data Access Interface）**：由数据访问类实现。它指定了用例交互器完成工作时需要如何访问数据。

> 注意，这些接口被视为属于用例层，完整规约了系统的核心逻辑——_其他一切_都只是细节！

### 11.3.4. 实体（Entities）

实体层包含程序的"模型"或临时信息存储。"企业业务规则"（最高层策略）由这一层强制执行。例如，ACORN 背后的"企业"是多伦多大学，如果多伦多大学规定学生在同一学期不能注册超过 7 门课程，那么 Student 实体类就会强制执行这条规则。

- **实体（Entity）**：存储程序基础构件相关信息的类。例如，汽车租赁应用可能有表示单个汽车（Car 类）、单个租用者（Renter 类）等的类。

## 11.4. CA 引擎的实际运行

下面的图与我们刚刚看到的相同，但标注了控制流。

> 先自行追踪一遍控制流，然后再阅读下面更详细的示例追踪。

![](images/Clean_Architecture_Typical_Scenario_Diagram_with_flow.png)

### 第1步：View → Controller
用户与 View 交互，触发一个事件，用户输入通过该事件传递给 Controller。

### 第2步：Controller → 输入边界（由用例交互器对象实现）
Controller 将用户输入打包到一个输入数据对象中，并通过输入边界传递（即调用输入边界接口中定义的、以输入数据对象为参数的方法）。

### 第3步：用例交互器 → 数据访问接口（由数据访问对象实现）
作为工作的一部分，用例交互器可能需要通过数据访问接口读取数据。例如，它可能根据 Controller 传入的输入数据对象中的信息，请求获取一个实体对象。

### 第4步：数据访问 → 数据库
数据访问对象执行必要的操作，从实际存储数据的数据库中读取数据。

### 第5步：用例交互器 → 实体
一旦用例交互器从数据访问对象获得所需的实体对象，它就使用这些实体的方法来完成工作。

> 此时，用例交互器可能需要额外调用数据访问接口的方法，例如保存信息。即，第3步至第5步的顺序和重复次数将取决于用例交互器的具体细节。

### 第6步：用例交互器 → 输出边界（由 Presenter 对象实现）
用例交互器完成工作后，创建一个输出数据对象，并通过输出边界传递（即调用输出边界接口中定义的、以输出数据对象为参数的方法）。

### 第7步：Presenter → ViewModel
最后一步是 Presenter 从输出数据对象中提取信息，并更新 ViewModel 以反映用户交互的结果。View 会收到这一变化的通知，并可以相应地更新自身。至此，交互完成，用户将看到更新后的 View。

> 思考问题：注意我们从未向 Controller 返回任何值。这对输入边界所定义方法的返回类型意味着什么？

## 11.5. 主组件（Main Component）

我们现在了解了涉及的类和接口的结构，以及信息如何在 CA 引擎中流动，但我们究竟是如何构建 CA 引擎的？

答案很简单——我们仔细地创建所有实例并将它们连接在一起。我们一直使用的 UML 类图省略了方法和实例变量存在的细节，但根据上述交互的描述，我们可以推断出哪些类必须包含对其他类（或接口）的引用。例如，为了使上述第1步得以发生，View 必须持有一个 Controller 实例。类似地，对于第2步，Controller 必须持有一个输入边界类型的引用，以便执行用例交互器。

程序的主组件负责 CA 引擎的初始组装，包括创建各种类的实例并将它们适当地连接在一起。这是程序中我们选择如何配置系统的地方——例如，选择将哪个接口实现实际插入系统。这正是我们设计价值的体现。例如，如果我们在测试代码时，可以使用一个专门为测试边缘情况设计的数据访问接口实现，而非在生产版本中实际使用的实现。

这意味着我们可以为不同目的创建不同的主组件，有时称为"驱动程序（driver）"。
