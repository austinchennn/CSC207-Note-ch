# 第9章：设计原则

## 9.1. 软件设计原则的演进

1935 年，Alan Turing 提出了计算机的定义，包括一个计算的数学模型（图灵机）。

20 世纪 50 年代，"结构化编程"语言被发明出来——在此之前，程序员用汇编语言或机器码编写程序（最早期的程序甚至需要拨动开关来设置比特位！）。FORTRAN、COBOL 和 LISP 的发明是为了让人类能更好地表达算法。这些编程语言都需要编译器或解释器自动将程序翻译成机器码。

20 世纪 60 年代，[Margaret Hamilton](https://en.wikipedia.org/wiki/Margaret_Hamilton_(software_engineer)) 创造了"软件工程"这一术语。她是麻省理工学院仪表实验室软件工程部的主任，被誉为[可靠软件设计的先驱](https://web.archive.org/web/20101124231727/http://www.nasa.gov/home/hqnews/2003/sep/HQ_03281_Hamilton_Honor.html)。

20 世纪 70 年代，IBM 等公司开始研究如何设计大型软件系统，70 年代中期，耦合与内聚被认定为良好软件设计的基础概念。

直到 90 年代末，软件开发项目经常严重超预算、延期交付。人们为寻找解决这一问题的"银弹"付出了大量努力，发明了许多工具和流程。其中一些有所帮助，但始终没有找到真正的银弹。

## 9.2. SOLID 原则

在那数十年间，人们发明/发现了一系列软件设计原则，使软件更易于理解、维护，并在添加新功能时更灵活。其中五条原则被广泛认为是良好设计的基础，它们都有助于保持低耦合和高内聚。2000 年代，Robert Martin 提议将它们整合为一个缩写：**SOLID**。

这些 [SOLID 原则](https://en.wikipedia.org/wiki/SOLID)在业界被广泛使用，在软件实习面试中也经常被提及。

在整个 CSC207 课程中，我们将使用这些原则来分析软件设计选择。

1. _单一职责原则（Single Responsibility Principle, SRP）_：一个类应该只有一个发生变化的原因
2. _开闭原则（Open/Closed Principle, OCP）_：一个类应该对扩展开放，对修改关闭
3. _里氏替换原则（Liskov Substitution Principle, LSP）_：子类应该可以替换其基类
4. _接口隔离原则（Interface Segregation Principle, ISP）_：程序员不应该被迫编写或依赖他们不使用的接口方法
5. _依赖反转原则（Dependency Inversion Principle, DIP）_：高层模块不应该依赖低层模块，两者都应该依赖抽象

### 9.2.1. SRP

> 一个类应该只有一个发生变化的原因

这条原则常被误解为"每个类只能有一个职责"，但这种简化并没有完整地体现该原则的含义。

SRP 的真正含义是："将因相同原因而改变的事物聚合在一起；将因不同原因而改变的事物分离开来。"（Robert Martin，https://blog.cleancoder.com/uncle-bob/2014/05/08/SingleReponsibilityPrinciple.html）

那么，谁导致了变化？是**参与者（actor）**！

_参与者_是程序的用户、利益相关方——或一组这样的人——甚至是一个自动化流程。

Robert Martin 进一步用以下方式优雅地描述了这条原则：

"这条原则是关于人的。……当你编写一个软件模块时，你要确保当有变更请求时，那些变更只能来自单一的一个人，或者更确切地说，来自代表单一、明确定义的业务功能的紧密耦合的一群人。你想要将模块与整个组织的复杂性隔离开来，并设计你的系统，使每个模块只负责（响应）一个业务功能的需求。"（Robert Martin，同上）

Robert Martin 还指出，SRP 本质上就是在描述高内聚和低耦合。其哲学含义当然不止于此，但这是基本思路。

SRP 通过确保每个类只因一个明确定义的原因而改变，帮助我们设计出更易于理解、测试和修改的类。

### 小示例

假设你有一个 `UserProfile` 类，同时负责显示用户信息和将其保存到数据库。这是两个不同的变化原因：UI 变化和数据库变化。SRP 建议将其拆分为两个类：一个负责显示逻辑，一个负责数据持久化逻辑。

### 9.2.2. OCP

> 一个类应该对扩展开放，对修改关闭

开闭原则意味着你应该能够在不修改现有代码的前提下为程序添加新功能。

例如，IntelliJ 以及许多游戏和其他应用程序都有"插件"或"扩展"系统。编写插件时，你编写的代码会被主程序调用，但你_不会_直接编辑主程序的代码。相反，插件实现了一组主程序将会调用的方法。这就是开闭原则在实践中的体现。

这通常通过继承或接口来实现。在 Java 中，接口可以用于指定插件类需要实现的方法，这样主程序就不需要知道插件类的确切类型。

通常而言，为什么在添加新功能时修改现有代码被认为是不可取的？让我们来看一个例子。

### 示例：修改代码的风险

想象两个开发者 Alice 和 Bob 在项目的不同部分工作，两人都依赖一个共享的 Java 类。Bob 意识到他需要这个类的行为稍微改变一下，于是修改了它并让自己的部分正常工作。但当 Alice 更新代码时，她发现 Bob 的修改破坏了她的功能。

这种情况在协作开发中很常见。当我们修改现有代码来添加新功能时，存在破坏依赖该代码的系统其他部分的风险。这也使原本代码的意图更难理解，并可能引入难以追踪的细微 bug。

开闭原则鼓励我们通过设计——使新行为可以通过扩展现有代码（继承、接口或组合）来添加，而非修改它——来避免这种情况，从而保持现有功能的稳定性。

### 9.2.3. LSP

> 子类应该可以替换其基类

从形式上来说，如果 S 是 T 的子类型，那么 T 类型的对象可以被 S 类型的对象替换，而不会改变程序的任何期望属性。

在 Java 中，T 可以是类或接口；S 要么继承（extends）类 T，要么实现（implements）接口 T。

里氏替换原则是关于确保子类可以在其基类被使用的任何地方使用而不引发问题。大多数情况下这自然会发生，但要注意不要阻止在基类中正常工作的功能在子类中失效。一个重要方面是确保类型 T 被充分规约，以便使用它及其子类型的代码清楚地了解预期的行为。因此，在 Java 中编写有良好文档的接口是至关重要的。

### 示例：不会飞的鸟

例如，如果你有一个 `Bird` 类，其中有一个 `fly` 方法，然后你创建了一个 `Penguin` 子类，它重写了 `fly` 方法并抛出异常，那就违反了 LSP——因为企鹅不会飞。任何期望能对 `Bird` 调用 `fly` 的代码，在接收到 `Penguin` 时都会出错。

相反，你可以重新设计整体结构以避免这个问题。例如，你可以有 `Bird` 的子类 `FlyingBird`，然后让 `Sparrow` 继承 `FlyingBird`，让 `Penguin` 继承 `Bird`。

### 示例：正方形"是一种"矩形

从数学上讲，正方形是四条边都相等的特殊矩形，它对边长有额外的约束，这意味着正方形比矩形更受约束——或者说功能更少。

具体来说，可以想象将 `Square` 定义为 `Rectangle` 的子类，然后在计算 `Rectangle` 对象面积的程序中运行如下代码时遇到问题：

```java
public static void rectangleDemo(Rectangle r) {
    r.setWidth(2);
    r.setHeight(5);
    System.out.println(r.area()); // 应该打印 10
}
```

如果以 `Square` 类型的参数调用这个方法，代码肯定不会按照父类预期的方式工作。与此相关的是，正方形拥有 `setWidth` 和 `setHeight` 方法本来就没有意义。这个例子强调了如何设计抽象的重要性。尽管正方形在数学上是矩形，但这并不意味着使用"is-a"关系来建立继承关系就能得到良好设计的类集合。

> 这个例子在《整洁架构》一书中也有讨论。

### 9.2.4. ISP

> 程序员不应该被迫编写或依赖他们不使用的接口方法

接口隔离原则意味着任何程序员都不应该被迫依赖不相关的方法，也不应该被迫实现不相关的方法。让我们分别从实现者和客户端两个角度来探讨。

### 实现者视角

例如，如果你有一个 `Animal` 接口，包含 `eat`、`sleep` 和 `fly` 方法，那么实现 `Animal` 的 `Dog` 类就会被迫实现 `fly` 方法，而这对狗来说毫无意义。

相反，你可以将 `Animal` 接口拆分为更小的接口，比如 `Eater`、`Sleeper` 和 `Flyer`，每个接口只包含一两个方法。这样，`Dog` 类就只需要实现 `Eater` 和 `Sleeper` 接口。

> 随着 Java 引入默认方法，现在可以提供默认行为。因此，在上面的例子中，另一种方法是让 `fly` 默认什么也不做，这样只有当子类与飞行相关时才需要实现它。关于 Java 中 interface 关键字的有趣读物可以参考[这里](http://blog.cleancoder.com/uncle-bob/2015/01/08/InterfaceConsideredHarmful.html)。

### 客户端视角

当你是某个类或接口的客户端时，你不希望依赖与你目的无关的功能。设想这样一种情况：你的类依赖一个提供若干方法（`m1`、`m2`、`m3`）的 `Service` 接口。如果你的类只使用 `m1`，那么对 `m2` 或 `m3` 的任何修改——即使与你的需求无关——仍然会导致你的类需要重新编译，甚至重新测试。在大型系统中，这种重新编译时间的增加会拖慢团队迭代开发、添加新功能的速度。

这在你的代码与你并不关心的系统部分之间制造了不必要的耦合。ISP 告诉我们，通过设计更小、更专注的接口，只暴露每个客户端实际需要的内容，来避免这种依赖。

> 这一客户端视角在《整洁架构》教材中也有更详细的描述。

### 9.2.5. DIP

> 高层模块不应该依赖低层模块

这里的"模块"指的是一个类或一组紧密相关的类，例如可能存在于一个 `.java` 文件中的那些。

在构建复杂系统时，程序员往往倾向于先定义_低层_类，然后再构建直接使用这些低层类的_高层_类。这里的低层指的是实现的细节，这些细节往往是易变的、可能随时发生变化的，它们对应于外部依赖。相对地，高层指的是我们程序中真正重要的部分——即我们试图实现的核心逻辑。

但这种方式不够灵活！如果需要替换一个低层类怎么办？高层类中的逻辑就需要随之修改——这正是高耦合的表现。

为了避免这类问题，我们在低层类和高层类之间引入一个抽象层。如果能确保在低层与高层之间设计一个稳定的接口，就可以在代码中建立一条清晰的边界。这意味着程序中的依赖都将指向抽象。只要接口保持不变，就可以独立地在接口两侧进行开发！

### 示例：经理与员工

一家公司由经理和员工构成。代表公司结构的代码中有管理 `Worker` 的 `Manager`。假设公司正在重组，引入了新类型的员工，他们希望代码更新来反映这一变化。

你目前的代码中有 `Manager` 类和 `Worker` 类，`Manager` 类中有一个或多个以 `Worker` 实例为参数的方法。

现在出现了一种新的_超级员工_，其行为和特性与普通员工不同，但两者都有"做工作"的概念，因此定义了一个新的 `SuperWorker` 类。我们的系统大致如下：

## 9.3. 练习：Java 中 SOLID 的示例

建议你思考目前所学到的 Java 工作方式——Java 的哪些语言特性与 SOLID 原则相符？

例如，Java 要求在子类构造函数中调用父类构造函数，这与哪些原则相符？泛型如何体现这些原则？`private` 关键字呢？将接口用作引用类型的能力呢？

每当你遇到新的代码或 UML 类图时，都应该开始从 SOLID 原则的视角来评估其设计。
