# 第10章：分层架构入门

"低耦合、高内聚"是软件开发行业中最重要的概念之一。它是组织代码的指导原则，使程序的不同部分便于独立理解、测试和修改。

为了实现这一目标，软件通常被组织成多个层，每层有各自明确的职责——例如管理用户界面、处理数据持久化，或实现程序的核心逻辑。这些层有助于减少程序各部分之间的依赖，使系统更易于维护和演化。

但我们究竟在将什么组织成层？

计算机程序可以被看作一组**策略（policies）**——规定如何将输入转换为输出的规则。如果有帮助的话，可以将其理解为关于数据在系统中不同节点的格式要求的一组规则。在计算机科学术语中，策略提供了一组不变量、前置条件和后置条件。

策略存在于不同的**层级（levels）**：

- **高层策略**远离输入/输出，代表应用程序的核心逻辑。
- **低层策略**接近输入/输出，处理 UI 交互、文件访问或网络通信等事务。

策略的层级指的是其与输入输出的距离。按层级分离策略有助于将程序的核心逻辑与其与外部世界交互的技术细节隔离开来。

以下是两个高层策略的例子：

- 当用户借阅图书时，该图书不能被其他用户预约。
- 当客户从其高端银行账户中取款时，允许最多 $1000 的透支。

低层策略可能描述用户如何触发事件，以及数据如何存储在数据库中。

在分层架构中，目标是保持高层策略独立于低层策略。例如，在图书馆系统中，一条策略可能规定：如果有另一个用户预约了某本书，则不能续借该书。这是关于系统应该如何行为的高层决策，与用户如何与程序交互或数据如何存储无关。用户点击"续借"按钮，还是系统从数据库中查询信息，是各自独立的关注点。高层策略本身保持不变。这种分离提高了可测试性，降低了耦合，使系统更易于推理。

了解了分层架构如何自然地按层级分离策略后，让我们更近距离地看看程序运行时的行为。

## 10.1. 事件驱动编程：对象的网络

现代程序通常响应用户动作——点击、按键和其他交互。这些被称为**事件**，响应这些事件的程序称为**事件驱动**的。

当一个事件发生时，一个叫做**事件监听器（event listener）**的方法被触发。该方法接收一个包含用户交互详情的事件对象，例如点击了哪个按钮或按下了哪个键。监听器随后调用其他方法来处理事件，在对象间形成一个方法调用链。

这个调用链形成了一个**对象的网络**，每次方法调用代表一条连接。在运行时，调用栈追踪这些方法调用，堆（heap）存储涉及的对象。调用栈底部的方法通常是事件监听器，其余的调用栈反映了由该事件触发的逻辑流。

为了支持新的用户交互，我们编写新的事件监听器并定义它们应该如何响应，将用户输入与驱动应用程序的逻辑连接起来。

## 10.2. 层与耦合

软件开发者花了数十年来完善代码组织方式。一种被广泛采用的策略是将程序分层，每层有特定的职责——例如管理用户界面、处理数据存储或实现核心逻辑。这种分离有以下几个好处：

- 一层的变化不太可能影响其他层，降低了 bug 风险，使系统更易维护。
- 通过将相关代码分组在一起，有助于避免 Git 合并冲突。
- 尽量减少开发者修改时需要接触的文件数量，降低引入错误的可能性。
- 一层内的代码往往遵循一致的模式，例如 UI 类通常包含与用户交互相关的事件监听器方法。

### 层层嵌套

**模型（Model）**是你选择在程序中表示数据的方式，例如创建一个 `Book` 类，用 `Book` 对象来表示系统的当前状态。这类对象通常称为**实体（entities）**。

模型是一个重要的概念，因为实体代表了思考所操纵数据的方式。特别重要的是（出于可测试性和可维护性的考虑），模型不应该对 UI 层或数据库层有任何依赖。这意味着实体不能引用你在这些层中定义的任何类。

> 回顾问题：实体代表哪个层级的策略？

以下是几种_架构模式_，即思考程序组织方式的框架。如果你开始关注软件设计博客，会遇到这些模式。以下大致按其流行时间的先后顺序排列：

* 模型-视图-控制器（MVC）
* 模型-视图-呈现器（MVP）
* 模型-视图-视图模型（MVVM）
* 端口与适配器架构
* 洋葱架构
* 六边形架构
* 整洁架构（本课程将重点介绍）

它们都旨在保持低耦合和高内聚，都有在不引入耦合的情况下跨越层的技术。它们对"模型"的定义以及事件处理方式有细微差别，但其代表问题域数据的核心概念是共同的。

### 所有架构共有的方法

你需要编写一个事件监听器方法来处理用户交互，这个方法必须在 UI 层。

事件监听器方法将调用控制器/呈现器/视图模型层中的方法，后者再调用操纵问题域数据的方法。在某处，会有代码从数据库、文件或网络中保存和读取数据。

## 10.3. 模型-视图-控制器（MVC）架构模式

以下是一个典型的 MVC 思考过程：

![模型-视图-控制器信息流](images/MVCPackt.png)

* 用户与视图（View）交互，触发一个事件

* Controller 操纵模型（Model）
* Model 从数据库中获取必要的数据
* Model 调用 View 对象中的方法通知有更新
* Controller 决定显示哪个界面
* View 调用 Model 上的 getter 方法并更新 UI

## 10.4. 模型-视图-呈现器（MVP）架构模式

MVC 架构模式是最早在软件行业被广泛采用的，但 View 与 Model 之间的循环依赖和紧密耦合使其难以测试和维护。MVP 架构模式正是为了解决这些问题而被开发出来的。

你会注意到，Presenter 在 MVP 中扮演的角色与 MVC 中的 Controller 大致相同，但 View 和 Model 互不依赖。

![MVP.png](images/MVP.png)

* 用户与视图（View）交互，触发一个事件
* Java 系统调用 View 对象中的监听器方法
* View 立即调用呈现器（Presenter）中的方法来处理事件
* Presenter 操纵模型（Model）
* Model 从数据库中获取必要的数据
* Model 调用 Presenter 对象中的方法通知有更新
* Presenter 调用 View 中的方法来更新用户界面

## 10.5. 更细粒度的分层架构

其他架构——端口与适配器、洋葱、六边形和整洁架构——都是同一主题的变体。它们都有一个代表问题域数据和逻辑的核心层，以及围绕核心层处理用户交互、数据库、文件和网络的其他层。

核心层通常称为_领域（domain）_层，包含实体和业务逻辑。业务逻辑是实现问题域规则的代码，例如计算学生的 GPA 或确定用户是否有权访问某资源。

简而言之，整洁架构由四个层组成：

1. **实体（Entities）**：代表问题域数据和逻辑的核心层（最高层策略）
2. **用例（Use Cases）**：包含应用特定业务规则的层（中层策略）
3. **接口适配器（Interface Adapters）**：包含将数据从用例和实体最方便的格式转换为外部层（如用户界面）最方便的格式的代码的层（较低层策略）
4. **框架和驱动（Frameworks and Drivers）**：包含与用户以及任何数据库进行外部交互的代码的层（最低层策略）

> 在《整洁架构》一书中，我们在上面非正式称呼的"实体"和"用例"层使用的术语是"企业业务规则"和"应用业务规则"。

在这里，Model 位于实体层，Controller/Presenter/ViewModel 代码位于接口适配器层，View 和数据库都位于框架和驱动层，而用例层包含特定于单次用户交互的对象，每个对象都有一个 Controller 调用的方法，以尽可能多地处理用户交互。

我们将在下一章更详细地探讨整洁架构，因为它是本课程重点关注的架构。
