# 第0章：Git 入门

在本课程及未来的编程项目中，你将使用版本控制系统来管理代码。Git 是一种常见的版本控制系统，也是本课程重点介绍的工具。

## 0.1. 什么是版本控制？

版本控制系统正如其名：一种让我们管理代码版本的方式，用于追踪所做的修改、控制各个版本的修订记录。

设想一下，当你与他人（或在多台机器上与自己）协作完成一个编程项目时，你会怎么做：发送最新版本代码的 `.zip` 压缩包？把需要修改的代码片段发给持有"主版本"的人？还是使用类似 Google Docs 的协作软件，在同一个地方同步更新代码？

上述方案都有各自的缺陷：要么难以追踪当前版本或找出每步的修订记录，要么难以独立工作。一旦出错需要回滚，要么得找一个旧版本文件，要么得手动还原——极为繁琐！

版本控制没有这些缺陷：每个人都在各自独立的版本上工作，版本控制系统会在需要时将所有人的修改合并起来，并持续追踪每次修订。

## 0.2. 版本控制的工作原理

在版本控制系统中，存在一个**主（master）仓库**：保存所有文件的最新版本。人们通过**克隆（clone）**仓库来获取自己的本地副本，并在本地独立工作。

当某人完成修改并希望将其反映到主仓库时，他们会将更改**推送（push）**进去。同样，任何想要获取仓库最新版本的人都可以通过**拉取（pull）**来同步更改。

版本控制还有许多其他功能，以上只是最核心的概念和最简单的使用方式。

## 0.3. Git

Git 是本课程中使用的具体版本控制系统。

在 Git 中，主仓库也称为 **origin**。我们使用 `git clone <url>` 命令来获取仓库的本地副本——这类似于下载所有文件，但额外提供了通过 `git status` 追踪哪些文件被修改的功能。

### 0.3.1. 向 origin 提交修改

在完成文件的修改、添加和删除后，我们通常希望将这些变更同步到主仓库。为此，使用 `git add <文件>` 将所有已修改且希望更新到主仓库的文件添加到暂存区。

接下来运行 `git commit -m "<提交信息>"`，描述所做的更改（例如：`git commit -m "修复了 X 中的一个 bug"`）。最后，通过 `git push` 将更改推送到仓库！

因此，提交修改的一般工作流程如下：
- `git status`：查看本地副本中哪些文件发生了变化
- `git add <文件>`：将想要同步到 origin 的文件"暂存"起来
- `git commit -m "<提交信息>"`：将更改保存到本地仓库，并附上描述信息
- `git push`：将更改推送到 origin 仓库

此工作流有一个注意事项：如果主仓库中存在你本地尚未获取的新变更，你需要先**拉取**这些修订。

### 0.3.2. 拉取新版本

在与仓库协作时，你通常需要将他人的更改同步到本地仓库。只需运行 `git pull` 命令即可。

此时，你可能需要做一些额外的工作：对本地更改进行添加/提交，或手动处理合并冲突。

### 0.3.3. 分支

有时你希望在一个独立的仓库上开发某个完整的功能：在功能完成之前，直接提交到主分支可能并不合适，因为你只希望在功能完成后再合并。

这种情况下，最好创建一个新的**分支（branch）**：这是从主仓库派生出的一个独立副本，你可以在其中自由提交更改，而无需担心影响主分支，直到你准备好合并为止。

要创建并切换到新分支，可使用 `git checkout -b <分支名>`。这是 `git branch <分支名>` 和 `git checkout <分支名>` 的缩写：前者创建新分支，后者切换到该分支。之后的提交将进入 `<分支名>` 而非 origin。

要切换分支，只需使用 `git checkout <分支名>` 即可。

当最终要将分支合并到主仓库时，切换回主分支后运行 `git merge <分支名>`。不过更推荐的方式是创建一个 [Pull Request](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-a-pull-request)，让其他人能够查看和审核你的修改再合并。
