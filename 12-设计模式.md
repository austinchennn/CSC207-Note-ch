# 第12章：设计模式

学完本章后，你应该熟悉：
- 什么是设计模式，
- 设计模式的三大类别，
- 以及每个类别中两个示例设计模式的面貌。

## 12.1 什么是设计模式？

**设计模式**是对常见编程问题的流行解决方案的描述。如果实现了一个不够理想的解决方案，我们称之为**反模式（anti-pattern）**。如果代码体现了反模式，可以通过重构来实现设计模式，从而改善代码设计。

这些模式描述的是代码的结构，而不是细节；它们是传达设计思路的手段，不针对任何特定的编程语言。因此，UML 图自然可以用来解释设计模式的结构。

在 CSC301（软件工程导论）和 CSC302（大型软件系统工程）中你将学到更多设计模式，但本课程将重点介绍其中六个。

### 12.1.1. 与设计原则的关系

正如我们刚刚学到的整洁架构一样，设计模式本质上只是底层原则的应用，例如"低耦合、高内聚"和 SOLID 原则。

### 12.1.2. 类比：数学公理与定理

在数学中，_公理_是被假定为真而无需证明的基础真理。在软件设计中，设计原则（如 SOLID）是我们遵循以构建"良好"系统的基础准则，这些原则通常相当通用。

> 例子：
> - 公理：过两点恰好有一条直线。
> - 原则：一个类应该只有一个发生变化的原因。

在这两种情况下，公理和设计原则都是我们推理的起点。

在数学中，定理是从公理推导出的已证结果。在软件中，设计模式是将设计原则应用于常见问题而形成的既定解决方案，这些模式比其底层原则更具实际可操作性，不那么抽象。

> 例子：
> - 定理：三角形的内角和为 180°。
> - 模式：观察者模式解决了当一个对象改变时通知多个对象的问题。

设计模式建立在设计原则之上，就像数学定理建立在公理之上一样。因此，有些程序员根本不认为有必要学习这些模式，因为只要遵循设计原则，他们终将得到遵循该模式的代码。不过，设计模式确实带给我们一样东西：一种与其他程序员沟通更复杂设计思路的共同语言。

回到我们的类比：如果有人已经通过仔细、严格地应用一组公理证明了一个定理，你还会重复这项工作吗——还是直接使用他们的结论并将其应用于手头的任务？

> 练习：在阅读每个设计模式时，思考如何用一条或多条原则来"推导"出每个模式。

设计模式的概念最早由"四人帮"（Gang of Four）于 1995 年出版的书中提出。原书描述了 23 个模式：11 个_行为型_、5 个_创建型_和 7 个_结构型_模式。此后又有更多模式被加入，其他作者也写了更多相关书籍。

下面我们将了解这三大类模式，并分别考察每类中的两个示例。对于每个模式，我们将重点关注常见的编程问题是什么，以及模式如何解决该问题。

## 12.2. 创建型模式（Creational Patterns）

创建型模式聚焦于对象的创建。这些模式有助于避免常见的陷阱，如紧密耦合、构造函数过度使用，以及引入系统中硬依赖的不灵活的实例化逻辑。

从根本上说，创建型模式解决的是：`new MyClass(...)` 调用应该在代码的什么地方出现，以及创建的对象如何连接在一起。

> 换句话说，这些模式解决的问题是：_谁负责创建系统中的每个对象？_

工厂（Factory）和建造者（Builder）模式是创建型模式的两个代表性例子。

### 12.2.1. 工厂（Factory）

**问题：** 你需要创建都是同一个抽象的子类或实现类的对象，但不想让代码依赖具体的类名或构建方式。

**解决方案：** 使用一个特殊的"工厂"对象或方法，由它决定创建哪个类并返回一个可以直接使用的实例。

工厂模式有多种变体，但它们都共享相同的基本思路：

1. 第一种变体是**简单工厂（Simple Factory）**模式，_使用（通常是静态的）方法根据输入参数创建对象_。它可以产生共享同一接口或超类的不同具体类型。
2. 第二种变体是**工厂方法（Factory Method）**模式，_使用继承让子类决定实例化哪个具体类_。
3. 第三种变体是**抽象工厂（Abstract Factory）**模式，_使用组合将实例化逻辑委托给另一个对象_。这会创建一系列一个或多个相关对象。

> **注意：** 我们将重点介绍简单工厂和抽象工厂。

### 简单工厂示例

如果你的程序需要 `Shape` 对象，但不需要知道它们是矩形、圆形、三角形还是将来可能添加的其他形状，你可以使用 `ShapeFactory` 类来实例化这些类：

```java
public class ShapeFactory { 
    public Shape getShape(String shapeType) {
        switch(shapeType) {
            case "Rectangle": 
                return new Rectangle(); 
            case "Circle": 
                return new Circle(); 
            case "Square": 
                return new Square();
            default:
                throw new IllegalArgumentException(
                        "Wrong shape type: " + shapeType
                );
        }
    }
}
```

用户随后可以通过输入字符串来创建形状：

```java
ShapeFactory shapeFactory = new ShapeFactory();
Shape shape = shapeFactory.getShape(userInput);
// 对形状做一些操作...
```

> 注意：你有时会看到变体，其中 `getShape` 方法被定义为 _static_，此时代码简化为：
>
> `Shape shape = ShapeFactory.getShape(userInput);`
>
> 因为我们根本不需要创建 `ShapeFactory` 的实例！

### 抽象工厂示例

让我们用抽象工厂改写同一个例子。使用简单工厂时，`ShapeFactory.getShape` 方法内部有基于请求形状类型的条件逻辑。添加新形状时，就需要修改这个方法。

基于这一观察，抽象工厂可以将这种条件逻辑完全移出该方法。这意味着用户将在创建工厂_之前_指定要创建的形状类型。

首先需要一个定义"形状工厂"含义的抽象类：

```java
public abstract class ShapeFactory {
    public abstract Shape getShape();
}
```

然后为每种想要创建的形状定义子类：

```java
public class RectangleFactory extends ShapeFactory {
    public Shape getShape() {
        return new Rectangle();
    }
}

public class CircleFactory extends ShapeFactory {
    public Shape getShape() {
        return new Circle();
    }
}

public class SquareFactory extends ShapeFactory {
    public Shape getShape() {
        return new Square();
    }
}
```

客户端代码需要使用所选的工厂来创建它想要的 `Shape`：

```java
public class Main {
    public static void main(String[] args) {
        ShapeFactory factory = new CircleFactory();  // 可以根据用户输入来选择
        Shape shape = factory.getShape();
        // 对形状做一些操作...
    }
}
```

当然，如果基于用户输入决定使用哪个工厂，那么简单工厂版本中的条件逻辑又会出现，只是出现在代码的不同位置。最值得注意的是，一旦确定了使用哪个工厂，代码的其他任何地方就不再有条件逻辑了。

### 简单工厂与抽象工厂的比较

以下是简单工厂和抽象工厂关键方面差异的快速总结：

| 方面         | 简单工厂                           | 抽象工厂                             |
|------------|----------------------------------|-------------------------------------|
| 创建逻辑     | 全部在一个类中                     | 分散在各子类中                       |
| 添加新类型   | 添加新的 if 语句（修改现有代码）    | 添加新的工厂子类（扩展现有代码）     |
| 客户端使用   | 客户端通过 String 参数指定类型     | 客户端选择工厂                       |

### 12.2.2. 建造者（Builder）

**问题：** 创建一个有许多部件或选项的复杂对象会使你的代码凌乱、难以阅读。

**解决方案：** 使用一个逐步"建造"对象的建造者，一步步地组装对象，最终交付成品。

回顾 Hatchimal 的 `toString` 方法：

```java
public String toString() {
    return new StringBuilder()
             .append("Hatchimal{")
             .append("name='").append(name).append('\'')
             .append(", rarity=").append(rarity)
             .append('}')
             .toString();
    }
```

这遵循一个模式：
- 创建一个"建造者"对象（此处为 `StringBuilder`）
- 逐步向其添加部件（此处通过多次调用 `append()`）
- 获取最终产品（此处通过调用建造者的 `toString()` 方法）

你可能还从我们之前用于进行 API 调用的 `Okhttp` 代码中认出了这个模式：

```java
Request request = new Request.Builder()
                .url(String.format("%s/grade?username=%s", API_URL, username))
                .addHeader(TOKEN, getAPIToken())
                .addHeader(CONTENT_TYPE, APPLICATION_JSON)
                .build();
```

这再次展示了相同的顺序：创建建造者，执行一系列步骤来定制正在创建的对象，然后最终调用以返回对象。

> IntelliJ 有一个重构功能，可以[用建造者替换构造函数](https://www.jetbrains.com/help/idea/replace-constructor-with-builder.html)。

## 12.3. 行为型模式（Behavioural Patterns）

行为型模式聚焦于对象如何交互和通信。它们解决的问题是：职责和行为如何在系统中的对象之间分配，以及这些对象如何协作完成任务。我们将以策略（Strategy）和观察者（Observer）模式作为行为型模式的两个代表性例子。

### 12.3.1. 策略（Strategy）

**问题：** 你有多种算法来完成同一任务，并希望能轻松地在它们之间切换。

**解决方案：** 将每种算法实现在各自的类中，并使它们可以互换，这样就可以在不改变使用它的代码的情况下更换算法。

例如，考虑一个 `Map` 类，它需要一个 `getDirections` 方法，该方法应根据是驾车还是乘坐公共交通返回不同的 `String`：

```java
public class Map {

	private DirectionGenerator dg;

	public Map(DirectionGenerator dg) {
	    this.dg = dg;
	  }

	public getDirections() {
	    dg.getDirections();
	   }
}
```

我们将 `DirectionGenerator` 接口称为策略，并让 `DrivingDirections` 和 `TransitDirections` 类实现它。当 `Map` 类被实例化时，它被传入一个实现了策略的对象。

### 策略模式的 UML 类图

以下是策略模式的 UML 图：

> - 设置使用哪种策略通常通过构造函数（如示例中）或 setter（如 UML 类图中）来完成。
> - 策略往往对应于在多个上下文中都可能有用的代码。

### 12.3.2. 观察者（Observer）

**问题：** 程序的多个部分需要在一个对象改变时作出反应，但你不希望这些部分与被观察的对象之间存在紧密耦合。

**解决方案：** 定义一对多的关系：当被观察对象改变时，所有相关对象都会收到通知以便作出响应。被观察对象无需了解观察它的对象的任何具体信息。

例如，我们附加到 `JButton` 组件上的 `ActionListener` 对象：当按钮被点击（事件）时，监听器的 `actionPerformed` 方法被调用（反应）。

### ViewModel 示例

还记得整洁架构的 View 部分吗？每个 View 都有一个对应的 ViewModel 存储要显示的数据。当 ViewModel 中的数据改变时，View 需要更新自身以反映新数据。这可以通过观察者模式实现：每个 View 将观察其 ViewModel。

### 观察者模式的 UML 类图

以下是观察者模式的 UML 类图：

![](images/observer.png)

Observable 的职责是在自身改变时通知所有 Observer。在我们的例子中，ViewModel 是 Observable，View 是 Observer。

> 在 Java 中，这通常通过让 Observable 使用 `java.beans.PropertyChangeSupport` 对象来实现，该对象管理 `PropertyChangeListener` 对象（即 Observer）的列表，这样你就不必自己编写管理 Observer 列表的代码了。
>
> 当 Observable 的状态改变时，它调用其 `PropertyChangeSupport` 对象的 `firePropertyChange` 方法，后者依次调用每个 `PropertyChangeListener` 的 `propertyChange` 方法。

### 代码示例

```java
public class LoginView extends JPanel implements ActionListener, PropertyChangeListener {
    
    // 其他方法未展示。
    
    public void propertyChange(PropertyChangeEvent evt){
        // 响应变化事件。
        final LoginState state = (LoginState) evt.getNewValue();
        setFields(state);  // 该方法将根据新状态中的数据更新 View 的字段。
    }
}
```

以下是一个简单的 ViewModel，使用 `PropertyChangeSupport` 在状态改变时通知其 Observer。在登录示例中，这发生在 Presenter 更新完 ViewModel 之后：

```java
public class ViewModel<T> {
    private final PropertyChangeSupport support = new PropertyChangeSupport(this);
    
    public void addPropertyChangeListener(PropertyChangeListener listener) {
        this.support.addPropertyChangeListener(listener);
    }

    // 当 Presenter 更新完 ViewModel 时调用此方法。（注意它委托给一个辅助方法。）
    public void firePropertyChanged() {
        this.support.firePropertyChange("state", null, this.state);
    }

    public void firePropertyChanged(String propertyName) {
        this.support.firePropertyChange(propertyName, null, this.state);
    }
    
}
```

## 12.4. 结构型模式（Structural Patterns）

结构型模式聚焦于类和对象如何组合形成更大的系统。与创建型模式（处理_对象如何创建_）和行为型模式（处理_对象如何交互及共享职责_）不同，结构型模式解决的是_组件如何连接和组织_的问题。它们通常解决这样的问题：为复杂子系统提供简化、统一的接口，或适配不兼容的接口。我们将以适配器（Adapter）和外观（Façade）模式作为结构型模式的两个代表性例子。

### 12.4.1. 外观（Façade）

**问题：** 一个子系统向客户端暴露了许多类和操作，迫使客户端处理不必要的复杂性。

**解决方案：** 创建一个面向客户端的简单类（即_外观_），为常见任务提供方法，同时隐藏子系统的复杂细节。

例如，你可能还记得 SOLID 幻灯片"单一职责原则"部分中的 `EmployeeFacade` 类，它包含以下类型的变量：
- 负责计算员工薪资的 `EmployeeCalculator`
- 负责报告员工工时的 `EmployeeReporter`
- 负责将更新的员工信息发送到数据库的 `EmployeeSaver`

当你调用 `employee.calculatePay()` 时，执行的是以下代码。注意它如何使用适当的私有变量将工作委托给这三个对象之一：

```java
public class EmployeeFacade {

    private PayCalculator payCalculator;
    private HourReporter hourReporter;
    private EmployeeSaver employeeSaver;
    
    public float calculatePay() {
       return this.payCalculator.calculatePay();  // 注意这里是单行委托
    }
    
    // 其他方法也在这里，结构相同，委托给合适的对象。
}
```

### 12.4.2. 适配器（Adapter）

**问题：** 两个组件因为接口不匹配而无法协同工作。这在你需要复用一个为不同上下文设计的现有类时经常发生。

**解决方案：** 引入一个_适配器_——一个在预期接口与现有接口之间进行转换的类。适配器暴露客户端代码期望的接口，并在内部将调用转发给遗留类，根据需要转换输入和输出。这可以通过以下方式实现：

- **委托（Delegation）**：适配器持有对遗留类的引用，并将工作委托给它。
- **继承（Inheritance）**：适配器继承遗留类，并添加代码以与预期接口保持一致。

> 类比：想想 USB-C 转 HDMI 适配器。许多笔记本电脑可以通过 USB-C 输出视频，但你的显示器只接受 HDMI 输入。适配器既不改变笔记本电脑也不改变显示器——它只是在两个接口之间进行转换，使它们能够协同工作。

### 代码示例

假设我们有一个名为 `Ticket` 的类，最初是为一个彩票程序编写的。如果你的彩票号码与随机选出的号码匹配，你就赢了。

现在假设我们想在一个电影院售票程序中复用这个 `Ticket` 类。原始类不支持座位号，所以我们需要对它进行适配。以下是使用委托和继承两种版本的适配器模式的实现方式：

### 使用委托

```java
public class MovieTicket {

    private Ticket ticket = new Ticket(); // 被适配的类
    private String seatNumber;

    public MovieTicket(String seatNumber) {
        this.seatNumber = seatNumber;
    }

    public void sellTicket() {
        // 将工作委托给 Ticket 类并添加额外行为
        ticket.sell();
        System.out.println("Seat: " + seatNumber);
    }
}
```

这里，`MovieTicket` 通过持有一个 `Ticket` 实例并将调用委托给它来充当适配器。我们在不修改原始 `Ticket` 类的情况下添加了新功能（座位号）。

### 使用继承

```java
public class MovieTicket extends Ticket {

    private String seatNumber;

    public MovieTicket(String seatNumber) {
        super(); // 初始化 Ticket 部分
        this.seatNumber = seatNumber;
    }
    
    public void sellTicket() {
        // 复用父类的 sell 方法并添加额外行为
        super.sell();
        System.out.println("Seat: " + seatNumber);
    }
}
```

这里，`MovieTicket` 通过继承来适配原始的 `Ticket` 类。

## 12.5. 总结

下表简要总结了本章介绍的三类设计模式：

| **模式类别** | **关注点**         | **核心问题**                                         | **代表模式**       |
|--------------|-------------------|-----------------------------------------------------|--------------------|
| **创建型**   | 对象创建           | 谁负责创建系统中的每个对象？                         | 工厂、建造者       |
| **行为型**   | 对象交互与通信     | 职责和行为如何在对象间分配，对象如何协作完成任务？   | 策略、观察者       |
| **结构型**   | 对象与类的组织     | 如何连接和组织对象以协同工作？                       | 适配器、外观       |
